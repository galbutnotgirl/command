Background Architecture

Maintainer-oriented overview of how custom actions route captured content through local Claude or Codex CLI runs.

Provider routing

  • Claude uses claude -p and /skill syntax.
  • Codex uses codex exec -, -i image attachments, and $skill syntax.
  • Schema v2 keeps legacy Claude CLI settings and adds separate provider configurations.
  • Codex defaults to read-only; workspace-write requires explicit selection.

Stack decision

Command is a native macOS menu-bar app, not Electron. The app keeps its native capture layer for hotkeys, selected text, screenshots, popup text, voice, clipboard attribution, and settings, then reuses the Electron-free Node core from the imported capture-and-handoff app.

Imported pieceNative equivalent
src/main.js tray and global shortcutsagent/main.swift Carbon hotkeys and menu bar.
src/capture/* capture helperssend-to-claude.sh and native capture stages.
Electron settings and text-entry UINative Background Settings sheet and popup trigger panel.

Kept core modules include settings, prompt, runner, submit, submissions, and paths. The preserved vendor contract lives at vendor/claude-command-capture/docs/HANDOFF.md.

Architecture

  1. Global hotkey fires for custom action trigger.
  2. Native agent captures selected text, screenshot, popup text, or voice transcript.
  3. Action prompt renders with {selection}, {file}, context, and source.
  4. Vendored Node core submits rendered prompt to configured Claude CLI.
  5. Submission record and logs are written locally.

Data layout & contract

Base directory is ~/Library/Application Support/claude-command. Override with CLAUDE_CAPTURE_HOME.

settings.json keeps imported schema fields such as skill, compatibility prompt fields, cli.command, cli.baseArgs, cli.extraArgs, cli.cwd, and notifications. Electron hotkeys remain in the schema for compatibility but native hotkeys live in ~/.claude/state/command-hotkeys.json.

Source mapping nuance

Selected-text runs record source: "selection" even when fresh clipboard fallback supplies the text. Screenshots record source: "screenshot". The imported core still accepts text and clipboard for other callers.

Using it

Create a custom action in Settings -> Shortcuts -> Custom Actions, choose Background delivery, optionally set a skill, save, then add one or more triggers: Selected text, Screenshot, Popup, or Voice.

Shared CLI config lives in Settings -> Command History -> Background Settings and writes ~/Library/Application Support/claude-command/settings.json.

Settings

Shared CLI settings live in Settings -> Command History -> Background Settings. Per-action behavior lives in Settings -> Shortcuts.

SurfaceOwns
ShortcutsAction prompt, triggers, delivery, destination, skill.
Background SettingsCLI command, working directory, extra args, notifications.

Records

Submission records live under ~/Library/Application Support/claude-command/submissions/. Command history retention defaults to 7 days.

Native UI (agent/Handoff.swift + MenuBar.swift)

  • Background Settings sheet: shared CLI command, working directory, extra args, and notifications.
  • CustomActionTextEntryPanel: popup trigger panel; Command-Return submits, Esc closes.
  • Command History tab: foreground commands plus background submissions, filters, retry, retention, and mark failed.

Result Contract

If last non-empty stdout line from background run is KEY=value, Command stores it as result and shows it in notification plus Command History.

TASK_ID=abc123

Background Custom Actions — building a structured background-prompt flow

A background Custom Action is a normal Custom Action with delivery set to Background. Instead of pasting the rendered prompt into Claude, Command pipes it to claude -p and leaves a submission record. This mirrors a Shortcuts automation that hands text to a shell script, except the trigger, prompt, history, retry, and retention live in Command.

  1. Add a Custom Action in Settings -> Shortcuts -> Custom Actions.
  2. Set delivery to Background. Add selected-text, screenshot, popup, or voice triggers after saving.
  3. Write an explicit prompt contract: parse the input, do the work, then emit one machine-readable final line.
  4. Set shared CLI flags in Settings -> Command History -> Background Settings when a headless run needs allowed tools or noninteractive permission behavior.
  5. Bind a hotkey or use the action from the menu. Check result, logs, retry, and retention in Command History.
